Memory-leak review: client-windows + client-macos

Reviewed 2026-07-07. Scope: do any objects/resources stay alive or accumulate when the client processes run for weeks or months? All long-lived components were read in full: loops, timers, process lifecycles, HTTP clients, SQLite usage, AppKit windows, closures.

Verdict: no true runtime memory leaks in either client. Two genuine unbounded-growth issues were found (one in both clients, one macOS-only), plus small bounded nits.

Finding 1 FIX Sent events are never purged from the local SQLite buffer (both clients)

MarkSent / markSent stamps rows with sent_at_utc, but nothing ever deletes them — there is no DELETE anywhere in either buffer:

  • client-windows/internal/buffer/buffer.go (MarkSent, line ~111)
  • client-macos/Sources/RootaikaMac/EventBuffer.swift (markSent, line ~157)

The heartbeat produces at least ~1,440 events/day (more during active use), so the database file and the idx_events_unsent index grow without bound for the device's lifetime. Growth is slow (megabytes per month) but it is the one place where objects live forever on a long-running client.

Fix (one statement per client)

After a successful markSent (or on open), purge old sent rows:

DELETE FROM events
WHERE sent_at_utc IS NOT NULL
  AND sent_at_utc < <now - 7 days>

A 7-day retention is safe: idempotency dedup happens server-side via the per-event UUID, and the same UUID is never re-enqueued locally (the agent generates a fresh UUID per observation; a failed post is retried as a new event on the next tick).

Finding 2 FIX macOS Updater.download creates a URLSession per call and never invalidates it

client-macos/Sources/RootaikaMac/Updater.swift (line ~70) creates URLSession(configuration:) inside download. A URLSession keeps its internal resources (delegate queue etc.) alive until finishTasksAndInvalidate() / invalidateAndCancel() is called — this is a documented leak pattern.

Normally download runs once and the process restarts after a successful update. But in the broken-release state (download/apply fails → 30-minute cooldown → retry) it leaks ~48 sessions/day for as long as the bad desired_version stays configured.

Fix

let session = URLSession(configuration: config)
defer { session.finishTasksAndInvalidate() }

NetworkBoardClient and DaemonClient sessions are created once per process and are fine as-is.

Bounded nits NO ACTION REQUIRED

  • macOS log growth (disk, not memory): MacDebugConsole.log (DebugConsole.swift:47) mirrors every line to stderr even when debug mode is off — ~3–4 lines per 5 s tick, roughly a gigabyte per year in the launchd log file, which launchd does not rotate.
  • Windows sound process never Wait()ed: playLoop (controller_windows.go:373) kills the MP3-loop process but never reaps it; the handle is released only by the GC finalizer. Bounded to one per lock cycle.
  • Windows warn-cycle context: the context.WithCancel created in agentapp.go:103 is cancelled only at unlock, not when the countdown completes naturally (go vet's lostcancel would flag it). Bounded to one per lock cycle, does not accumulate.

Checked and clean

Windows client

  • All loops use time.NewTimer + Stop (no time.After leaks).
  • Per-call api.New shares http.DefaultTransport: connections are pooled and bounded.
  • All response bodies, rows cursors, prepared statements and transactions are closed, including error paths.
  • agentrunner and the lock overlay's Wait goroutine reap child processes (no zombies).
  • The loopback HTTP server is shut down via context.

macOS client

  • Lock/warning NSWindows are torn down on unlock and references released.
  • keepOnTopTimer is invalidated and uses [weak self]; key monitor removed; AVAudioPlayer stopped and nil'd; cursor hide/unhide balanced.
  • warningTask is cancelled/nil'd on both the unlock and natural-completion paths.
  • Debug console buffers are capped (5,000 backlog lines / 200,000 text-view characters).
  • EventBuffer finalizes every prepared statement, including error paths.
  • AgentHTTPServer cancels the connection on every terminal path.
  • The daemon's infinite Task.detached loops retain self deliberately (process-lifetime singleton).
