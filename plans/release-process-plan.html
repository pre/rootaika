Release process: container build + GitHub Releases

Automate publishing the Windows client (rootaika.exe) and its SHA-256 digest as a GitHub release. The admin drafts and publishes the release (generated notes, reviewed in the browser); the build runs in a container that is bit-for-bit identical in GitHub Actions and on a local machine, and the artifacts are built and attached by a separate machine account so no personal credentials appear in CI. Builds on the OTA design in plans/auto-update-suunnitelma.html.

StatusImplemented (build.sh, Makefile, release.yml, README). The bot PAT and the RELEASE_BOT_TOKEN secret are configured manually on GitHub.

Decisions

Topic Decision Rationale
Build container Pinned official image, e.g. golang:1.24.5-bookworm. No custom Dockerfile. The build needs nothing beyond the Go toolchain (pure Go, no cgo). The image tag pinned in the Makefile is the single source of truth; the same image runs locally and in CI.
CI / local parity The workflow runs the build with the exact same make docker-build target used locally (docker run on the runner host, not a container: job). Literally one command shared by both environments. Keeping the job on the host also keeps the preinstalled gh CLI available for publishing.
Reproducible build Add CGO_ENABLED=0, -trimpath, -buildvcs=false to build.sh. Same commit + same image → bit-identical rootaika.exe → same SHA-256. Anyone can rebuild at the tag and verify the release asset, which strengthens the OTA trust anchor (the admin-entered SHA-256).
Publisher identity Machine account (e.g. rootaika-bot) as a collaborator with write access on pre/rootaika; its PAT stored as the RELEASE_BOT_TOKEN Actions secret. Personal-account credentials never appear in CI. The release itself is authored and published by the admin (who creates the draft); the bot only builds, attaches the assets, and appends the SHA-256 triple to the notes. The assets must still land in pre/rootaika because the client pins owner/repo at compile time (client-windows/internal/version/version.go).
Trigger release: published. make github-release VERSION=vX.Y.Z creates a draft release pinned to the current HEAD (--target), with notes generated from conventional commits since the previous v* tag (grouped Features / Fixes / Tests / Chores / Other). Pressing Publish release in the browser creates the tag and fires the workflow. The admin reviews and publishes; the tag still follows the OTA URL scheme releases/download/{tag}/rootaika.exe. Pinning the target commit at draft time keeps the notes and the tagged code in sync even if main moves before publishing. A bare tag push does nothing — one publishing path only.
Release contents rootaika.exe, rootaika.exe.sha256, install.ps1; the workflow appends the version triple for the admin UI to the release notes after the build (idempotent on re-runs). Same asset set as the retired scripts/release.sh; the .sha256 file works with sha256sum -c. The digest exists only after the build, so it cannot be part of the reviewed draft notes.
Server registration Stays manual: paste the triple into the admin UI (Versiot). The LAN server is unreachable from CI, and manually entering the SHA-256 is the deliberate trust anchor of the OTA design.

Changes

  1. client-windows/scripts/build.sh — add CGO_ENABLED=0, -trimpath, -buildvcs=false to the build, and write dist/rootaika.exe.sha256 (sha256sum format) next to the binary.
  2. client-windows/Makefile — pin GO_IMAGE and add one pattern target that runs any make target inside the image: 
    GO_IMAGE ?= golang:1.24.5-bookworm

docker-%:
	docker run --rm -v "$(CURDIR)":/src -w /src \
	  -v rootaika-gomod:/go/pkg/mod \
	  $(GO_IMAGE) make $* VERSION="$(VERSION)"
    This gives make docker-build VERSION=v1.3.0 and make docker-test for free. The named volume caches Go modules between runs.
  3. .github/workflows/release.yml — new workflow: 
    name: release
on:
  push:
    tags: ["v*"]
permissions:
  contents: read        # GITHUB_TOKEN is not used for publishing
jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: make -C client-windows docker-test
      - run: make -C client-windows docker-build VERSION="$GITHUB_REF_NAME"
      - name: Publish release
        env:
          GH_TOKEN: ${{ secrets.RELEASE_BOT_TOKEN }}
        run: |
          SHA256=$(cut -d' ' -f1 client-windows/dist/rootaika.exe.sha256)
          gh release create "$GITHUB_REF_NAME" \
            client-windows/dist/rootaika.exe \
            client-windows/dist/rootaika.exe.sha256 \
            client-windows/scripts/install.ps1 \
            --repo pre/rootaika --title "$GITHUB_REF_NAME" \
            --notes "desired_client_version = $GITHUB_REF_NAME
          client_artifact_name   = rootaika.exe
          client_sha256          = $SHA256"
  4. README.md — replace the “publish a release from a Linux box” section with the draft-and-publish flow, the local verification recipe, and the one-time bot setup. scripts/release.sh is removed: a release it published would now also fire the workflow, producing a second, conflicting asset upload. Manual recovery is make docker-build + gh release upload.
  5. client-windows/scripts/github-release.sh — creates the draft release: generates the grouped notes from git log, verifies HEAD is pushed and the tag is free, and prints the draft URL. --dry-run prints the notes without touching GitHub.

One-time bot account setup (manual)

  1. Create the machine account rootaika-bot (GitHub ToS permits machine accounts operated by a person).
  2. pre/rootaika → Settings → Collaborators → invite rootaika-bot with Write access.
  3. As the bot, create a PAT. Try a fine-grained PAT first; if pre/rootaika is not selectable as a resource (fine-grained PATs have historically not covered collaborator access to another user’s personal repos), fall back to a classic PAT with only the public_repo scope. Set expiry to ~1 year and note the renewal date.
  4. pre/rootaika → Settings → Secrets and variables → Actions → new secret RELEASE_BOT_TOKEN.
The workflow itself still runs inside pre/rootaika (free for a public repo). Only the publishing identity is the bot — the default GITHUB_TOKEN is restricted to read-only and never publishes.

Day-to-day release flow

make -C client-windows github-release VERSION=v1.3.0
# open the printed draft link, review the notes, press "Publish release"
# publishing creates the tag; Actions builds and attaches the assets and
# appends the triple to the notes; paste it into the admin UI (Versiot)
# and select the version for a test device first

Independent verification of a published release

git checkout v1.3.0
make -C client-windows docker-build VERSION=v1.3.0
cat client-windows/dist/rootaika.exe.sha256   # must equal the release asset digest

Verification of the implementation

  1. Run make docker-build VERSION=v0.0.1-test twice locally — the SHA-256 must be identical (reproducibility).
  2. Run make github-release VERSION=v0.0.1-test, publish the draft — the workflow attaches the assets and appends the triple; the locally rebuilt digest matches the asset. Delete the test release and tag afterwards.
  3. OTA end-to-end: register the triple, select it as a per-device override on one test machine, confirm the client updates.

Out of scope

  • Custom Dockerfile, GHCR images, server/macOS/waveshare release artifacts.
  • Code signing / attestation — the reproducible build + admin-entered SHA-256 covers the threat model; the exe stays unsigned exactly as today.
  • Changelog or version-bump tooling.
  • Automatic version registration into the LAN server.
When bumping the Go version later, update GO_IMAGE in the Makefile and go.mod together — reproducibility depends on both sides resolving the same toolchain.
