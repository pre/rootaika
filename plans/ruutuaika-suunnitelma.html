Suunnitelma

rootaika

Tavoitteena on toteuttaa Windows-pelikoneiden ruutuajan seurantaan LAN-verkossa toimiva järjestelmä, jossa Windows-pelikoneilla ajettava headless-client raportoi aktiivisen käytön Go-pohjaiselle HTTP-palvelimelle. Palvelin tallentaa tapahtumat SQLiteen ja tarjoaa web-käyttöliittymän toteuman seurantaan.

Poikkeamat suunnitelman ja toteutuksen välillä

Tätä taulukkoa ylläpidetään toteutuksen eri vaiheissa. Se kokoaa yhteen kohdat, joissa toteutus poikkeaa alkuperäisestä suunnitelmasta tai joissa suunnitelmaa on tarkennettu toteutuksen aikana. Tila kertoo, onko poikkeama korjattu, työn alla, vielä tekemättä, vai tietoisesti hyväksytty.

Nimeäminen ja repon rakenne

Palvelukokonaisuuden nimi on rootaika. Koodi toteutetaan monorepona git-repoon ~/dev/rootaika. Repon päätasolla on oma hakemisto kullekin pääkomponentille. 

rootaika/
  plans/
  server/
  client-windows/
  • server on hyvä nimi Go-pohjaiselle HTTP-palvelimelle, koska monorepon nimi antaa palvelulle jo tuotteen kontekstin.
  • windows on liian yleinen nimi client-komponentille: se voi tarkoittaa käyttöjärjestelmää, ikkunointia tai Windows-kohtaista apukoodia.
  • client-windows kertoo selvästi, että kyseessä on Windowsille toteutettu client. Jos myöhemmin tulisi muita clienttejä, nimeämistä voi jatkaa samalla kaavalla, esimerkiksi client-linux.

Windows-clientin sisällä voidaan erottaa service ja käyttäjäsessiossa ajettava agentti omiin alihakemistoihinsa, esimerkiksi client-windows/cmd/rootaika-service ja client-windows/cmd/rootaika-agent.

Rajaukset ja lähtökohdat

  • Kaikki laitteet ovat samassa LAN-verkossa.
  • Palvelin käyttää LANissa HTTP-yhteyttä ilman TLS/HTTPS-suojausta.
  • Ruutuaika seurataan ensisijaisesti laitekohtaisesti.
  • Jokainen laite voidaan liittää käyttäjään. Tulevaisuudessa saman käyttäjän useat laitteet voidaan laskea yhteen.
  • Alkuvaiheessa tuetaan yhtä aktiivista kirjautunutta Windows-käyttäjää per kone.
  • Ruutuaika tarkoittaa aikaa, jolloin kone ei ole idle-tilassa.
  • Idle-tilaan siirrytään oletuksena, jos hiirtä tai näppäimistöä ei ole käytetty 60 sekuntiin.
  • Idle-raja on konfiguroitava ja client cachettaa viimeisimmän configin paikallisesti.
  • Prosessin nimi riittää ensimmäiseksi aktiivisen ohjelman tunnisteeksi.
  • Ikkunaotsikoita, selainosoitteita tai muuta ylimääräistä yksityistä tietoa ei kerätä.
  • Kaikki ajat tallennetaan UTC-aikaleimoina. Esityskerroksessa voidaan käyttää Helsinki-aikavyöhykettä.
  • Kaikki raakatapahtumat säilytetään toistaiseksi pysyvästi ilman automaattista vanhenemista.

Tunnistettu rajoite: videon katselu voi näyttää idle-ajalta, jos hiirtä tai näppäimistöä ei käytetä. Tätä ei ratkaista ensimmäisessä versiossa.

Komponentit

1. Go HTTP Server

  • Vastaanottaa Windows-clienttien tapahtumabatcheja.
  • Tarjoaa client-configin ja komennot, kuten lock/unlock.
  • Tallentaa datan SQLite-tietokantaan.
  • Laskee raportit tapahtumista palvelinpäässä.
  • Tarjoaa web-käyttöliittymän toteuman seurantaan ja asetusten muokkaamiseen.
  • Ajetaan Linux-palvelimella Docker-kontissa.

2. Windows Service

  • Käynnistyy Windowsin mukana.
  • Säilyttää client UUID:n, Basic Auth -client-tunnuksen ja viimeisimmän configin.
  • Puskuroi eventit paikalliseen SQLite-tietokantaan.
  • Lähettää eventit palvelimelle ajoittain batcheina.
  • Pollaa palvelimelta configin ja pending-komennot.
  • Valvoo käyttäjäsession agenttia ja käynnistää sen uudelleen, jos agentti sammuu.
  • Ajetaan adminin asentamana servicena, jota ei-admin-käyttäjä ei voi normaalisti pysäyttää.

3. Windows User Session Agent

  • Ajetaan kirjautuneen käyttäjän sessiossa.
  • Ei näy taskbarissa eikä alt-tab-valikossa. Konsoli-ikkuna piilotetaan oletuksena.
  • Jos palvelinpäästä kytketty debug_mode on päällä, agentti näyttää konsoli-ikkunan, josta voi devauksessa seurata mitä tapahtuu.
  • Lukee idle-ajan Windows API:n GetLastInputInfo-kutsulla.
  • Lukee aktiivisen ikkunan GetForegroundWindow-kutsulla.
  • Selvittää prosessin PID:n GetWindowThreadProcessId-kutsulla.
  • Selvittää aktiivisen prosessin nimen PID:n perusteella.
  • Näyttää lukitustilassa vihreän fullscreen topmost -ikkunan, jossa lukee rootaika ja palvelimelta tuleva viesti.
  • Jos käyttäjä saa agenttiprosessin sammutettua, Windows Service havaitsee tilanteen ja käynnistää agentin uudelleen.

Autentikointi ja oikeudet

Järjestelmä käyttää HTTP Basic Authia kahdella roolilla:

Rooli Käyttö Oikeudet
admin Adminin web-kirjautuminen ja ylläpitotoiminnot. Voi muuttaa asetuksia, käyttäjiä, laitteita, kategorioita ja lock/unlock-komentoja.
client Windows-clienttien raportointi ja web-UI:n katselu. Voi lähettää tapahtumia, hakea client-configin, hakea komentoja ja katsella web-käyttöliittymää.

Jokainen client generoi itselleen pysyvän UUID-tunnisteen. Palvelin yksilöi laitteen tämän UUID:n perusteella ja hyväksyy tapahtumia clientiltä, joka läpäisee jaetun client Basic Auth -tunnuksen. Jos palvelin näkee uuden UUID:n ensimmäistä kertaa, se luo laitteen automaattisesti tilaan unassigned. Admin nimeää laitteen ja liittää sen käyttäjään web-UI:ssa.

Turvarajoite: yhteinen client-salasana ei estä clienttiä A raportoimasta clienttinä B, jos B:n UUID tiedetään. Lisäksi HTTP Basic Auth -tunnukset kulkevat LANissa ilman TLS-suojausta ja salasanat tallennetaan SQLiteen selväkielisinä. Nämä rajoitteet hyväksytään ensimmäisessä versiossa käsin konfiguroinnin yksinkertaistamiseksi.

  • Admin-toiminnot vaativat aina admin-tunnuksen.
  • Client-tunnuksella saa katsella koko web-UI:ta read-only-tilassa, mutta ei muuttaa asetuksia.
  • Clientin paikallinen config suojataan tiedosto-oikeuksilla niin, että ei-admin-käyttäjä ei voi muokata sitä.
  • Lapsen Windows-käyttäjätili pidetään ei-admin-tilinä.

Clientin tapahtumamalli

Client lähettää aikaleimattuja havaintoja aktiivisuuden tilasta. Lopullinen käyttöajan laskenta tehdään palvelimella vertaamalla saman laitteen peräkkäisiä havaintoja. Jokaisella tapahtumalla on oma UUID, jotta sama batch voidaan lähettää uudelleen ilman duplikaatteja.

  • Tapahtumatyyppi on activity_observed.
  • state kertoo havaitun tilan: active, idle tai locked.
  • process_name asetetaan vain, kun state on active.
  • occurred_at on UTC-aikaleima hetkelle, josta havaittu tila pätee laskennassa.
{
  "client_id": "2f0c70e7-70e5-4a1a-b9c4-5dd92dd9cf9b",
  "events": [
    {
      "event_id": "4c516bf0-a6c1-4d80-a227-f8022c5c8f3c",
      "type": "activity_observed",
      "occurred_at": "2026-06-11T12:00:00Z",
      "state": "active",
      "process_name": "steam.exe",
      "sequence": 12345
    }
  ]
}

Raportointirytmi

  • Idle-raja: oletuksena 60 sekuntia, palvelimelta konfiguroitava.
  • Foreground-prosessin tarkistus: esimerkiksi 1-5 sekunnin välein.
  • activity_observed syntyy säännöllisesti sekä heti, kun tila tai aktiivinen prosessi vaihtuu.
  • Eventtien lähetys: batchinä esimerkiksi 1 minuutin välein.
  • Configin ja komentojen polling: esimerkiksi 30 sekunnin välein. Config sisältää myös debug_mode-lipun.
  • activity_observed lähetetään heti, kun tila tai aktiivinen prosessi vaihtuu, ja lisäksi heartbeat-välein, vaikka mikään ei muuttuisi.
  • Jos serveri ei vastaa, eventit jäävät paikalliseen SQLite-puskuriin ja lähetetään myöhemmin.

Raportit voidaan laskea aluksi suoraan events-taulusta. Jos laskenta hidastuu tai kyselyistä tulee monimutkaisia, lisätään myöhemmin aggregaattitaulu kuten usage_segments tai daily_usage. Raakatapahtumille ei tehdä automaattista poistopolitiikkaa ensimmäisessä versiossa.

Käyttöajan laskenta palvelimella

  • Palvelin järjestää laitteen activity_observed-havainnot occurred_at_utc-ajan ja sequence-arvon mukaan.
  • Jos edellinen havainto on state = active, aika seuraavaan havaintoon lasketaan edellisen havainnon process_name-ohjelmalle.
  • Jos edellinen havainto on state = idle tai state = locked, aikaa ei lasketa ruutuajaksi.
  • Kun aktiivinen prosessi vaihtuu, client lähettää uuden activity_observed-havainnon uudella process_name-arvolla.
  • Pitkä havaintokatkos katkaistaan konfiguroitavalla max_countable_gap_seconds-rajalla, jotta sleep, sammutus tai verkko-ongelma ei tuota virheellistä käyttöaikaa.
  • Viimeistä avointa havaintoa ei tarvitse tallentaa palvelimella tilana. Raporttikysely voi laskea sen mukaan vain, jos havainto on riittävän tuore.

Tämän mallin etu on, että palvelin ei tarvitse erillistä start/stop-tilakonetta. Se koostaa käyttöajan aikaleimatuista havainnoista, ja laskentalogiikkaa voidaan korjata palvelinpäässä ilman client-päivitystä.

Web-käyttöliittymän MVP

  • Tänään per laite: aktiivinen kokonaisaika, ohjelmakohtainen erittely, viimeksi nähty ja lukitustila.
  • Viikkonäkymä: päiväkohtaiset summat per laite.
  • Kuukausinäkymä: päiväkohtainen tai viikkokohtainen yhteenveto.
  • Laitteet: nimi, UUID, käyttäjäliitos, viimeisin yhteys ja laitteen poisto.
  • Käyttäjät: käyttäjien luonti ja nimeäminen.
  • Asetukset: idle-raja, upload-väli, polling-väli, maksimilaskentaväli, debug-tila ja rekisteröimättömien clientien debug-oletus.
  • Kategoriat: prosessinimipohjaiset kategoriat, esim. pelit, selain ja muu.
  • Admin-toiminnot: lock/unlock, asetusten muokkaus ja laite-/käyttäjähallinta.

Lukitustoiminto

  • Admin luo web-UI:sta lock- tai unlock-komennon tietylle laitteelle.
  • Client hakee komennon pollingilla.
  • Windows Service välittää lukitustilan käyttäjäsession agentille loopback-HTTP:n (127.0.0.1) yli token-autentikoinnilla.
  • Agentti näyttää vihreän fullscreen topmost -ikkunan, jossa lukee rootaika ja palvelimelta saatu viesti.
  • Agentti palauttaa ikkunan etualalle, jos fokus vaihtuu.
  • Lukitus puretaan vain palvelimelta tulevalla unlock-komennolla.

Tämä on MVP-tason lukitus, ei Windowsin oma lock screen. Ei-admin-käyttäjälle se on kuitenkin käytännössä riittävä ensimmäiseksi toteutukseksi, kun service ja config on suojattu oikeuksilla ja service käynnistää agentin uudelleen tarvittaessa.

Clientin sammuttamisen vaikeuttaminen

  • Varsinainen valvova komponentti on Windows Service, ei käyttäjäsessiossa ajettava overlay-agentti.
  • Service asennetaan admin-oikeuksilla ja ajetaan esimerkiksi LocalSystem-tilillä tai erillisellä service-tilillä.
  • Ei-admin-käyttäjä ei saa oikeutta pysäyttää, poistaa tai muuttaa serviceä.
  • Windowsin service recovery asetetaan käynnistämään service uudelleen, jos se kaatuu.
  • Service tarkistaa säännöllisesti, että käyttäjäsession agentti on käynnissä oikeassa sessiossa.
  • Jos agentti puuttuu tai se on sammutettu, service käynnistää sen uudelleen.
  • Jos laite on lukitussa tilassa, uudelleenkäynnistetty agentti avaa vihreän topmost-overlayn heti uudelleen.
  • Clientin config, tunnukset ja paikallinen SQLite-puskuri suojataan ACL-oikeuksilla adminille ja service-käyttäjälle.

Prosessia ei piiloteta Task Managerista. Kovennus perustuu Windowsin normaaleihin oikeuksiin, serviceen, watchdog-valvontaan ja siihen, että lapsen käyttäjätili ei ole local admin.

Windows-toteutuksen huomioita

  • Windows-client voidaan toteuttaa Go:lla käyttäen golang.org/x/sys/windows-pakettia.
  • Go:lla onnistuvat idle-tunnistus, foreground window -tunnistus, prosessinimen selvitys, Windows Service, HTTP-client ja SQLite-puskurointi.
  • C# olisi luonteva erityisesti Windows UI- ja session-integraatioihin, mutta Go on realistinen valinta tämän rajauksen perusteella.
  • Client asennetaan MVP:ssä manuaalisesti ajettavalla PowerShell-asennusskriptillä admin-oikeuksilla.
  • Prosessia ei yritetä piilottaa Task Managerista. Sellaiset keinot menevät haittaohjelma-/rootkit-tekniikoiden puolelle.
  • Sen sijaan lapsen tili pidetään ei-admin-tilinä, service asennetaan admin-oikeuksilla ja service recovery käynnistää servicen uudelleen kaatumisen jälkeen.

Toteutusvaiheet

  1. Go HTTP Serverin runko: reititys, SQLite-yhteys, migraatiot, Basic Auth, HTTP ilman TLS:ää ja Dockerfile.
  2. Tietomalli: users, devices, events, config, commands, categories ja settings.
  3. Client API: batch events, config, commands ja command ack.
  4. Web-UI:n katselunäkymät: tänään, viikko, kuukausi ja ohjelmakohtainen erittely.
  5. Admin-UI: asetukset, käyttäjät, laitteet, kategoriat ja lock/unlock.
  6. Windows Service: UUID, config-cache, local SQLite queue, upload worker ja command polling.
  7. User Session Agent: idle-tunnistus, foreground process -tunnistus, eventtien välitys servicelle loopback-HTTP:n yli ja vihreä viestillinen overlay lukitustilassa.
  8. Windows-asennus: manuaalisesti ajettava PowerShell-skripti, joka asentaa servicen ja suojaa configin ACL-oikeuksilla.
  9. Server-laskenta: eventeistä käyttöjaksot ja raportit.
  10. Watchdog: service valvoo session-agenttia ja käynnistää sen uudelleen, jos käyttäjä sammuttaa sen.
  11. Lukitustoiminto: vihreä topmost fullscreen overlay viestillä ja unlock palvelimelta.
  12. Kovennukset: tiedosto-oikeudet, service recovery, retry/backoff, lokitus ja virhetilanteiden käsittely.

MVP-tavoite

Ensimmäisen toimivan version tavoite on: selaimesta näkee jokaisen Windows-koneen tämän päivän aktiivisen käytön, ohjelmakohtaiset ajat, viikko- ja kuukausiyhteenvedot sekä laitteen lukitustilan. Admin voi muuttaa asetuksia ja lähettää lock/unlock-komennon. Client toimii offline-tilanteessa puskuroimalla eventit paikallisesti ja lähettämällä ne myöhemmin.
