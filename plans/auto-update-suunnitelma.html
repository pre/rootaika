Plan

Windows client OTA auto-update

The Windows client updates itself over the air. The server declares a desired client version; the client notices the mismatch during its normal config poll, downloads a prebuilt binary from a public GitHub release, verifies it with SHA256, swaps itself, and restarts, all without rebooting Windows. The old process is stopped and the new one starts in its place.

Background and goal

Today the client is installed manually on each of the 5–10 home gaming PCs and has no update path: any fix or new feature means re-running install.ps1 on every machine. The goal is over-the-air updates driven by the server.

Constraints (confirmed)

  • No new API calls. The client reports its current version inside the existing GET /api/v1/client/config poll, and the server returns the desired version in the same response. This mirrors how status is already piggy-backed onto that poll.
  • GitHub owner/repo are fixed at compile time and are not server-controllable (security). Only the release tag, asset name, and SHA256 come from the server.
  • Single combined binary. Merge rootaika-service and rootaika-agent into one rootaika.exe with subcommands, so one file swap updates both processes.
  • Public repo (github.com/pre/rootaika) → anonymous HTTPS download, no token.
  • SHA256 integrity check before swapping the binary.
  • amd64 only. Windows amd64 is the only target.
  • Desired version is global with a per-device override, so one machine can be tested before the rest.
  • Update triggers on any difference (including a downgrade) → plain string inequality, no semver ordering.
  • The release is published from a local machine via a script. No GitHub Actions yet.
Why the service does the work: rootaika-service runs as LocalSystem, is the durable component, already polls the config endpoint, and controls the agent through its watchdog. It is the natural place to download, verify, and orchestrate the swap.

1. Combined binary (cmd/rootaika)

Replace the two cmd/ entrypoints with one dispatcher that runs from a single linked exe (linked -H=windowsgui as the agent already is; the service needs no console under the SCM).

  • New client-windows/cmd/rootaika/main.go dispatches on os.Args[1]: service → serviceapp.Run, agent → agentapp.Run, apply-update → updater.ApplyUpdate (the transient self-swap helper, see §5).
  • Move agent logic from cmd/rootaika-agent/main.go into a new internal/agentapp package (and its main_test.go → agentapp tests). The service already lives in internal/serviceapp.
  • Delete cmd/rootaika-service and cmd/rootaika-agent.
  • Debug console: the service must also attach/alloc a console in debug mode now that the binary is GUI-subsystem. Reuse internal/consolewin (already windows/other split) in serviceapp.Run, the same pattern the agent uses via console.SetVisible.

2. Version and fixed repo (internal/version)

New client-windows/internal/version/version.go:

package version

// const, NOT ldflags: the server can never change the download origin.
const GitHubOwner = "pre"
const GitHubRepo  = "rootaika"

// injected at build:
//   -X rootaika/client-windows/internal/version.Version=v1.2.0
var Version = "dev"

The download URL is built only from these constants plus the server-supplied tag and asset name: https://github.com/pre/rootaika/releases/download/{tag}/{asset} (works anonymously for public repos).

3. Poll carries version both ways

  1. Client → server: api.Client.FetchConfig (internal/api/client.go) gains a version argument, added as a &version= query param. pollOnce (internal/serviceapp/app.go) passes version.Version.
  2. Server records it: in handleClientConfig (server/internal/rootaika/handlers.go), after the existing status handling, record version via a new Store.RecordDeviceVersion. Add devices.last_version and last_version_at_utc columns with ensureColumn.
  3. Server → client: the config JSON map in handleClientConfig gains desired_version, artifact_name, and sha256. The client model.ClientConfig (internal/model/model.go) gains the matching fields.
These three update directives are transient. They must NOT be persisted to client.json → do not add them to ApplyServerConfig.

4. Server-side version config (global + per-device)

A version is a triple (version_tag, artifact_name, sha256) so a per-device override stays self-consistent with a different release.

  • Global (settings table, new keys): desired_client_version, client_artifact_name, client_sha256.
  • Per-device override (device_config, new columns via ensureColumn): desired_version, desired_artifact_name, desired_sha256 (all empty = inherit global).
  • Resolution in Store.ClientConfig: the effective triple is the per-device triple when desired_version != "", otherwise the global triple. Surface it on rootaika.ClientConfig (models.go) and emit it in the handler.
Refactor required: settingsFromDB and settingsFromTx (store.go) currently strconv.Atoi every settings row. Adding string-valued settings breaks this. Change both to read a key→string map, parse only the known integer keys into Settings, and assign the string keys directly. Add the string fields to the Settings struct (models.go) and to settingsFromForm + UpdateSettings.

Admin UI (settings_views.go)

  • Add the three global version fields to the settings form.
  • Add a compact per-device override form (3 inputs) in the devices table.
  • Show each device's reported last_version in a new column.
  • The per-device form posts to a new adminDeviceVersion case in handleAdmin, e.g. /admin/devices/{id}/version (follows the existing lock/assign/delete routing pattern).

5. Update orchestration (client)

New package client-windows/internal/updater. Pure parts are testable on Linux; OS calls are split with build-tag pairs.

  • updater.go (pure):
    • NeedsUpdate(current string, p Plan) bool → true when desired is non-empty and != current.
    • Download(ctx, p, destPath) builds the URL from version.GitHubOwner/Repo + p, streams the asset to destPath, and verifies SHA256. A mismatch is a terminal error and no swap happens.
  • apply_windows.go / apply_other.go: ApplyUpdate(args) is the detached helper:
    1. sc stop rootaika-service and wait for STOPPED,
    2. kill any running agent process,
    3. rename rootaika.exe → rootaika.exe.old, copy staged rootaika.update.exe → rootaika.exe,
    4. sc start rootaika-service (its watchdog respawns the agent from the new exe),
    5. best-effort delete rootaika.exe.old.
Self-replacement is safe: a running Windows exe can be renamed (not overwritten), and both processes are stopped before the swap. The detached helper is what swaps the service, since the service cannot replace its own running file.

Wiring in serviceapp.pollOnce

After applying config, if NeedsUpdate: Download to <installDir>/rootaika.update.exe, then launch the staged exe detached: 

rootaika.update.exe apply-update -target <exe> -service rootaika-service -config <cfg>

then initiate its own stop. Loop guard: remember the last failed (version, time) in stateStore and skip retrying the same failed version for ~30 minutes, so a bad release does not spin.

6. Watchdog uses the combined exe

internal/agentrunner/runner.go: when Path is empty, resolve to os.Executable() and run it with ["agent", "-config", cfgPath] instead of looking for rootaika-agent.exe. This guarantees both processes always come from the one on-disk file, so a single swap updates both.

7. Install / uninstall + release scripts

  • scripts/install.ps1: copy the single rootaika.exe; service binPath = "rootaika.exe service -config ..."; HKLM Run = "rootaika.exe agent -config ..."; launch rootaika.exe agent for the current session.
  • scripts/uninstall.ps1: unchanged service/run names; -Purge also removes rootaika.exe.old and rootaika.update.exe.
  • scripts/build.ps1: build one rootaika.exe with -ldflags "-H=windowsgui -X ...version.Version=$Version". This stays as PowerShell for building/launching on a Windows box; release publishing is the Linux release.sh below.
  • New scripts/release.sh v1.2.0 (Linux shell, not PowerShell): cross-compile with GOOS=windows GOARCH=amd64 go build -ldflags "-H=windowsgui -X ...version.Version=$1" -o dist/rootaika.exe ./cmd/rootaika, compute SHA256 (sha256sum), gh release create "$1" dist/rootaika.exe (public), then print the triple to paste into the admin UI: desired_client_version=v1.2.0, client_artifact_name=rootaika.exe, client_sha256=<hash>.

Reuse (do not write new)

  • The poll pipeline already piggy-backs device state via the status query param and returns a flat JSON config map → version travels the same route.
  • Schema additions use the existing ensureColumn migration pattern.
  • Snapshot/update of in-memory config uses the existing stateStore.
  • HTTP retry/backoff for transient failures already exists in internal/api/client.go; the download path can reuse the same retry stance (retry network/5xx, never 4xx, terminal on SHA256 mismatch).
  • Console show/hide in debug mode already exists in internal/consolewin with a windows/other split.

Verification

  • Unit (Linux, go test -race -cover ./... in both modules):
    • updater.NeedsUpdate truth table (empty desired, equal, different/downgrade).
    • updater.Download against an httptest server: correct URL from consts + plan, SHA256 match vs mismatch (mismatch = error, no file promoted).
    • Server: ClientConfig resolves the per-device triple over the global one; RecordDeviceVersion persists; settingsFromDB still parses ints after string settings are added; the handler emits the three fields.
  • Manual (real Windows, per the _windows.go convention in CLAUDE.md):
    1. Build v1 with release.ps1, install, confirm the device and version show in the admin UI.
    2. Build v2, gh release create, set the global triple in the admin UI.
    3. Within one poll interval the service downloads, verifies, swaps, and restarts; the admin shows v2; no reboot; the agent overlay/locking still work. Verify .old is cleaned up and the SCM service is healthy.
    4. SHA256 mismatch → the client stays on v1, logs an error, and backs off (no loop).
    5. A per-device override on one machine updates only it; clearing it falls back to global.
  • Cross-compile: GOOS=windows GOARCH=amd64 go build ./... in client-windows/.
